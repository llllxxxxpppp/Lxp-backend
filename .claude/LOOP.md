# LOOP.md

# 마스터 에이전트 작업 절차

## 0단계: 상태 확인 (매 세션 시작 시)
1. 주요 문서를 반드시 읽는다: `.claude/ARCHITECTURE.md`, `.claude/DOMAIN.md`, `.claude/TASK.md`(존재 시)
    - `.claude/task/`의 세부 파일은 이 시점에 읽지 않는다. 작업 선택 후 해당 도메인 파일만 읽는다.
2. 사용자 지시에서 대상 BC를 식별하고, TASK.md 상단의 BC 로드맵을 확인해 분기한다:
    - (a) TASK.md가 없다 → 사용자에게 BC 로드맵 작성이 필요함을 알리고,로드맵 초안(BC 목록·의존 방향·도메인 정의 완성 여부)을 제안한 뒤 사용자 확정을 받아 TASK.md 상단에 기록한다. 이후 (b)로 진행한다.
    - (b) 대상 BC의 도메인 정의가 미완(로드맵 기준) → 계획을 수립하지 않는다. "도메인 정의 완료가 선행되어야 함"을 보고하고, 정의가 완료된 다른 착수 가능 BC를 안내한다.
    - (c) 대상 BC가 정의 완료 + 계획 미수립 (요약표에 해당 BC 작업 없음) → 1단계로 (해당 BC만 계획).
    - (d) 대상 BC의 계획이 수립되어 있음 → 2단계로.
    - (e) 요약표에는 작업이 있는데 세부 파일이 없다 (또는 그 반대) → 불일치이므로 사용자에게 보고하고 지시를 받는다. 임의로 복구하지 않는다.
    - (f) 사용자가 전체 계획 수립을 지시한 경우 → 로드맵에서 도메인 정의가 완료(✅)된 모든 BC를 의존성 순서로 나열하고, 각 BC에 대해 1단계를 순차 수행한다.
        - BC 하나의 계획(도출→대조)이 끝날 때마다 중간 결과를 기록하고 다음 BC로 진행하며, 승인은 전체 완료 후 일괄로 받는다. (단, 사용자가 요청하면 BC별로 승인받는다.)
        - 정의 미완(⬜) BC는 계획하지 않고, 로드맵에 존재하는 것으로 대신한다. 건너뛴 BC 목록을 최종 보고에 명시한다.
        - 이 경우는 "사용자가 전체를 지시한 것"이므로 금지 사항의 'BC 임의 생성 금지'에 해당하지 않는다.
3. 사용자가 BC를 지정하지 않았으면, 로드맵과 요약표의 착수 가능(⚪) 현황을 보여주고 무엇을 진행할지 확인받는다.
2. task.md, 세부작업(경로: .claude/task/)이 존재하면 → 2단계(실행 루프)로 간다. task.md, 세부작업(경로: .claude/task/)이 없으면 → 1단계(계획 수립)를 먼저 수행한다.

## 1단계: 계획 수립 (task.md 생성 및 수정)
1. DOMAIN.md의 해당 BC 섹션과 ARCHITECTURE.md를 근거로, 해당 BC의 작업을 도출한다.
2. 작업 크기 제약:
   - 작업 1개 = 독립적인 완료 기준을 가질 수 있는 완결 단위이면서, 파일 수정 3~5개 이내
   - 작업 1개 = "테스트 작성 → 구현 → 리뷰 → 테스트 실행" 1사이클로 완료  가능한 크기 (산출물 기준 5천 토큰 미만)
   - 이 제약을 초과하는 작업은 반드시 더 작게 분할한다.
   - 예외: 기계적 횡단 변경(일괄 리네임, 의존성 추가 등)은 파일 수 제약을 초과할 수 있다. 단, 계획 시점에 "횡단 작업"으로 명시하고 사용자 승인을 받은 경우에 한한다.
3. 각 작업에 ID, 설명, 완료 기준(기계 판정 가능), 관련 규칙 위치, 의존성을 기재하고 의존성 순서로 정렬한다.
    3-1. 다른 BC에 대한 의존이 있는 경우 로드맵의 의존 방향을 기준으로 처리한다:
        - 의존 BC가 이미 계획됨 → 해당 BC의 구체 작업 ID를 의존성에 기재한다.
        - 의존 BC가 미계획 → 의존성에 "BC:[BC명]"으로 기재하고 상태를 🟠로 둔다. 그 BC의 계획이 수립되면 구체 작업 ID로 치환한다. 의존 BC의 계획을 임의로 생성하지 않는다.
        - 로드맵에 없는 BC 간 의존을 발견하면 로드맵 갱신을 사용자에게 제안한다.
        (임의로 로드맵을 수정하지 않는다)
4. 작업 목록을 `.claude/PROGRESS.md`와 대조하여 기존 진행분을 반영한다.
    - 이 대조는 작업 도출(1~3번)이 끝난 뒤에 수행한다. 작업 도출 중에는 PROGRESS.md를 참조하지 않는다.
    - 🟢 완료: PROGRESS.md에 완료를 뒷받침하는 명시적 기록이 있고, 실제 코드·테스트 파일의 존재를 확인한 경우에만
    - 🟡 보류(확인 필요): 기록이 모호하거나 부분 완료로 보이는 경우. 확인할 내용을 세부 파일에 기록한다
    - ⚪ 대기: 기록 없음
    - 🟢로 판정한 작업은 세부 파일 진행 기록에 근거를 병기한다. (근거: PROGRESS.md "기록 요약" + 확인된 파일 경로)
    - 추측으로 🟢를 부여하지 않는다. 근거가 불충분하면 🟡이다.
    4-1. 자리표시자 치환: 요약표 전체에서 의존성이 "BC:[지금 계획한 BC명]"인작업을 찾아, 이번에 생성한 구체 작업 ID로 치환한다.
        - 어느 작업 ID로 치환할지는 두 작업의 내용을 근거로 판단하고, 판단 근거를 해당 작업의 세부 파일 진행 기록에 남긴다.
        - 치환 후 의존성이 모두 충족된 🟠 작업은 ⚪로 갱신한다.
5. 생성한 TASK.md 요약표와 대조 결과(🟢/🟡 목록과 각 근거)를 사용자에게 보여주고 승인을 받는다.
   - 특히 🟢 판정 목록은 명시적으로 검토를 요청한다. 승인 전에는 2단계로 진행하지 않는다.

## 2단계: 실행 루프 (작업이 남아 있는 동안 반복)
1. TASK.md 요약표에서 의존성이 충족된 첫 번째 ⚪ 작업을 선택한다.
   - 사용자가 특정 작업을 지정하면 그 작업을 우선하되, 🟠(의존 미충족)이면 사용자에게 알리고 확인을 받는다.
   - 🟡 작업은 자동 선택하지 않는다. 사용자 지시가 있을 때만 착수한다.
   - 🔵 작업이 남아 있으면(이전 세션 중단 등) 임의로 이어받지 않고, 이어서 진행할지 처음부터 다시 할지 사용자에게 확인받는다.
2. 선점: 요약표에서 해당 작업의 상태를 🔵로 바꾸고 담당(세션 식별자)을 기록한 뒤 시작한다.
3. 선택한 작업의 컨텍스트를 구성한다:
   해당 도메인의 `.claude/task/task-[도메인명].md`에서 그 작업 섹션만 읽어
   작업 ID/설명/완료 기준 + 관련 규칙 원문 발췌 + 대상 파일 경로를 구성한다. (문서 전체를 전달하지 않는다. 이전 작업 이력을 전달하지 않는다.)
4. 파이프라인을 수행한다:
   4-1. [테스트 작성 에이전트] 완료 기준 기반 테스트 작성
   4-2. [코드구현 에이전트] 코드 구현
   4-3. [코드리뷰 에이전트] 리뷰
        → "수정 필요" 시: blocker/major 지적만 추려 4-2로 되돌린다
   4-4. [테스트 실행 에이전트] 테스트 실행
        → 실패 시 원인을 판별해 분기한다:
          - 원인 = 구현 코드 → 실패 정보와 함께 4-2로
          - 원인 = 테스트 코드 → 실패 정보와 함께 4-1로
   4-5. 재작업 한도: 동일 작업에서 되돌림이 총 3회를 초과하면
     요약표의 상태를 🔴로 변경하고, 세부 파일에 시도 내역을 기록한 뒤
     사용자에게 보고하고 루프를 중단한다.
5. 완료 처리:
   5-1. 완료 기준의 각 항목이 충족되었는지 최종 확인한다.
   5-2. 구현 결과를 사용자에게 보고하고 확인을 받는다.
   5-3. 세부 파일(task-[도메인명].md)의 해당 작업 진행 기록에 근거를 기록한다.(근거: 리뷰 승인 + 테스트 통과 N/N + 사용자 확인)
   5-4. TASK.md 요약표의 상태를 🟢로 변경한다. 5-3과 5-4는 반드시 함께 반영한다.
   5-5. 이 작업에 의존하던 🟠 작업들의 의존성이 모두 충족되면 ⚪로 갱신한다.
   5-6. 커밋 메시지를 제안하고, 사용자 확인 후에만 커밋을 생성한다.
     커밋에는 코드 변경 + 세부 파일 갱신 + TASK.md 갱신을 함께 포함한다.
6. 컨텍스트 사용량을 점검한다(컨텍스트 관리 섹션 참조).
   문제없으면 다음 작업으로 돌아간다(2단계 1번).
   대상 BC에 ⚪ 작업이 없으면 사용자에게 BC 완료를 보고한다.

## 3단계: 계획 변경 (실행 중 계획 수정이 필요할 때)

### 트리거
다음 중 하나가 발생하면 현재 작업을 일시 정지하고 이 단계를 수행한다:
- (T1) 서브 에이전트가 범위 밖 발견 사항을 보고함 (새 작업 필요)
- (T2) 작업의 완료 기준·대상 파일·분할이 실제와 맞지 않음이 확인됨
- (T3) 사용자가 계획 변경을 지시함 (작업 취소, 우선순위 변경, 정의 변경 포함)
- (T4) 🔴 처리된 작업의 재개를 위해 작업 재정의가 필요함

### 공통 원칙
- 마스터는 변경안을 제안만 하고, 사용자 승인 후에만 TASK.md와
  세부 파일에 반영한다. 승인 전에 계획 문서를 수정하지 않는다.
- 변경 이력은 해당 작업의 세부 파일 진행 기록에 남긴다.
  (변경일, 변경 내용, 사유, "사용자 승인" 표기)
- 진행 중이던 🔵 작업이 변경 대상이면, 변경 확정 후 컨텍스트를
  새로 구성해서 파이프라인을 처음부터 다시 시작한다.
  (변경 전 컨텍스트로 만든 중간 산출물을 이어 쓰지 않는다.)

### 변경 유형별 처리
1. 작업 추가 (T1):
   - 요약표에 ⚪ 행 추가 + 세부 파일에 섹션 추가.
   - 기존 작업과의 의존 관계를 검토해 필요 시 기존 작업을 🟠로 조정한다.
   - 현재 진행 중 작업과 무관하면, 기록만 하고 현재 작업을 계속한다.
2. 작업 수정 (T2, T4):
   - 세부 파일의 설명·완료 기준·대상 파일을 수정하고 변경 이력을 남긴다.
   - 분할이 필요하면: 원 작업 ID는 유지하되 하위 ID로 쪼갠다.
     (예: COURSE-03 → COURSE-03a, COURSE-03b. 의존성 참조가 깨지지 않도록
     원 ID를 삭제하지 않는다.)
   - 수정된 작업이 🟢였던 경우 → 아래 4번(완료 무효화)을 적용한다.
3. 작업 취소 (T3):
   - 요약표 상태를 ⚫(취소)로 변경하고, 사유를 세부 파일에 기록한다.
   - 행을 삭제하지 않는다. 취소된 작업에 의존하던 작업들의 의존성을
     재검토해 사용자에게 처리 방안을 확인받는다.
4. 완료 무효화 (정의 변경 등으로 🟢 작업이 영향을 받는 경우):
   - 영향 범위를 식별한다: 변경된 규칙을 "관련 규칙"으로 참조하는
     모든 작업(상태 무관)을 나열한다.
   - 영향받는 🟢 작업은 🟡(재검증 필요)로 변경하고 사유를 기록한다.
   - 재검증 방법(테스트 재실행만 / 리뷰 재수행 / 재구현)을 작업별로
     제안하고 사용자 승인을 받는다.
5. 도메인 정의 변경 (T3 중 DOMAIN.md 수정을 수반하는 경우):
   - DOMAIN.md 수정은 사용자가 직접 하거나 사용자 승인하에 수행한다.
   - 수정 후 반드시 4번(완료 무효화)의 영향 분석을 수행한다.
   - 필요 시 BC 로드맵의 갱신도 함께 제안한다.

### 복귀
변경 반영 커밋(계획 문서만의 커밋도 허용) 후, 2단계 1번(작업 선택)으로
돌아간다. 일시 정지했던 작업이 변경되지 않았다면 그 작업을 이어서 진행한다.

## 금지 사항
- 마스터는 코드를 직접 작성·수정하지 않는다.
- task.md에 없는 작업을 임의로 수행하지 않는다. 실행 중 새 작업이 필요해지면 task.md에 후보로 기록하고 사용자 승인 후 추가한다.
- 서브 에이전트 보고 없이 작업을 🟢로 변경하지 않는다.
- 세부 파일에 상태를 기록하지 않는다. 상태의 원본은 TASK.md 요약표뿐이다.
- PROGRESS.md는 1단계 4번(대조) 시에만 읽는다. 갱신하지 않으며, 상태 판단의 원본은 항상 TASK.md 요약표다.
- 사용자가 지시하지 않은 BC의 계획을 임의로 생성하지 않는다.
- 사용자 승인 없이 작업의 완료 기준·설명·의존성을 수정하지 않는다. (계획 변경은 3단계 절차를 따른다)

## 컨텍스트 관리

- 마스터는 작업 하나가 완료될 때마다(커밋 직후) 자신의 컨텍스트 사용량을 점검한다.
- 사용량이 약 70%를 넘었다고 판단되면, 다음 작업을 시작하기 전에 사용자에게 알린다:
  "컨텍스트 사용량이 70%를 넘었습니다. 지금이 정리하기 좋은 시점입니다.
   현재 상태는 TASK.md에 모두 기록되어 있어 /compact 또는 새 세션으로
   이어서 진행해도 손실이 없습니다."
- 알림 후 사용자의 선택을 기다린다. 임의로 정리하거나 계속 진행하지 않는다.
- 파이프라인 진행 도중(작업이 🔵인 상태)에는 컨텍스트 정리를 제안하지 않는다.
  반드시 작업 완료·커밋 후, 다음 작업 선택 전에만 제안한다.
- 예외: 진행 중 컨텍스트 고갈이 임박한 경우(자동 압축 경고 등),
  현재까지의 진행 내역(재작업 횟수, 마지막 완료 단계, 미해결 지적 사항)을
  해당 작업의 세부 파일 진행 기록에 즉시 기록한 뒤 사용자에게 알린다.
